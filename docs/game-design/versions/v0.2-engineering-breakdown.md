# 玩个球 V0.2 工程拆解

- 文档状态：工程拆解草案
- 创建日期：2026-05-02
- 上游文档：`v0.2-playable-loop.md`、`v0.2-content-tables.md`
- 目标：把 V0.2 内容表映射到当前代码结构，明确哪些数据放哪里、哪些规则由哪个系统负责、按什么顺序实现。

## 1. 当前工程现状

当前项目已经有一个可运行的类植物大战僵尸原型，主要模块如下：

| 模块 | 当前职责 |
| --- | --- |
| `packages/game-content/src/index.ts` | 当前运行时内容源：草坪、植物、僵尸、波次 |
| `packages/game-content/balance/` | 已存在 V0.2 风格的内容草案和 JSON 导出链路，但当前运行时还没有直接使用 |
| `packages/game-content/data/generated/` | `content:build` 生成的内容 JSON |
| `packages/game-core/src/types.ts` | 当前运行时状态类型：`Plant`、`Zombie`、`GameState`、`GameCommand`、`HudSnapshot` |
| `packages/game-core/src/engine.ts` | 游戏循环门面，调度 systems |
| `packages/game-core/src/commands.ts` | 处理选择、放置、开始波次、重置 |
| `packages/game-core/src/systems/` | 经济、植物攻击、投射物、僵尸移动与攻击、波次 |
| `apps/web/src/game/phaser/` | Phaser 场景、输入、渲染 |
| `apps/web/src/ui/` | React HUD、种子栏、资源、操作按钮 |

关键判断：

- 当前运行时仍是 `plant / sun / zombie` 语义。
- V0.2 目标是 `building / purchasePower / enemy / strategyCard` 语义。
- 为了降低第一轮改动风险，V0.2 实现可以先保留部分内部命名，再逐步重命名。

## 2. 迁移策略

推荐分两步走。

### 2.1 第一阶段：机制优先，命名兼容

目标是尽快跑通 V0.2 闭环。

允许：

- UI 里可以先把“植物”显示为“建筑”。
- `game-core` 内部可以短期保留 `Plant` 类型名，但字段和数据按建筑机制改。
- `SeedBank` 可以改名为 `BuildBar` 后置，先把按钮内容换成三类建筑。

必须改：

- 运行时内容要对齐 V0.2 内容表。
- 经济从 `sun` 语义改为购买力展示。
- 三类建筑机制必须成立：能量生产、方向炮塔、防御机关。
- 波次必须改为火山 5 波。
- 敌人必须支持普通、极速、重攻、飞天、Boss。
- 每波结束要生成 `BattleSnapshot`，并进入选卡阶段。

### 2.2 第二阶段：统一命名

V0.2 可玩后，再统一做命名清理。

建议重命名：

| 当前名 | 目标名 |
| --- | --- |
| `Plant` | `Building` |
| `PlantType` | `BuildingId` |
| `plantDefinitions` | `buildingDefinitions` |
| `plants` | `buildings` |
| `sun` | `purchasePower` |
| `SeedBank` | `BuildBar` |
| `Zombie` | `Enemy` |
| `zombies` | `enemies` |

这一步不建议和第一阶段混在一起做，否则容易把规则改动和命名改动搅在一起。

## 3. 目标数据流

V0.2 推荐数据流：

```text
docs/game-design/versions/v0.2-content-tables.md
-> packages/game-content/balance/*.ts
-> pnpm content:build
-> packages/game-content/data/generated/*.json
-> packages/game-content/src/index.ts 暴露运行时内容
-> packages/game-core 使用内容配置执行规则
-> apps/web 展示 HUD 和 Phaser 渲染
```

短期可以让 `src/index.ts` 直接导出 V0.2 内容，生成 JSON 作为策划和测试用的数据快照。

长期可以让 `game-core` 直接消费稳定的内容配置包，减少 `src/index.ts` 和 `balance/` 两套数据不一致。

## 4. 内容层拆解

负责目录：`packages/game-content`

### 4.1 类型定义

文件：

- `packages/game-content/balance/types.ts`
- `packages/game-content/src/index.ts`

需要新增或调整的核心类型：

```ts
type BuildingType = "energy" | "attack" | "defense";
type AttackDirection = "up" | "down" | "left" | "right";

type BuildingDef = {
  id: string;
  name: string;
  type: BuildingType;
  cost: number;
  hp: number;
  purchasePowerPerTick?: number;
  productionInterval?: number;
  attackPower?: number;
  attackInterval?: number;
  attackRange?: number;
  attackDirection?: AttackDirection;
  canBlockGround: boolean;
  canTargetGround: boolean;
  canTargetFlying: boolean;
  tags: string[];
  specialEffectHooks: string[];
  upgrade?: {
    cost: number;
    hpBonus?: number;
    attackPowerBonus?: number;
    attackRangeBonus?: number;
    purchasePowerPerTickBonus?: number;
  };
};
```

敌人类型要支持：

- `normal`
- `fast`
- `heavy_attack`
- `flying`
- `boss`

策略卡类型要支持：

- 救急
- 能量
- 攻击
- 防御
- 转向

### 4.2 内容配置

建议文件：

| 文件 | 内容 |
| --- | --- |
| `balance/buildings.ts` | `energy_core`、`arrow_turret`、`lava_wall` |
| `balance/enemies.ts` | 4 种基础敌人 + `volcano_core_beast` |
| `balance/waves.ts` | 火山 5 波 |
| `balance/cards.ts` | 15 张策略卡 |
| `balance/maps.ts` | 火山地图，baseHp 10，单局 2-5 分钟 |
| `balance/directorRules.ts` | 敌方 AI 下一波调整规则 |

当前已有 `balance/towers.ts`，有两种选择：

1. 保守方案：保留 `towers.ts` 文件名，但内容改成三类建筑。
2. 清晰方案：新增 `buildings.ts`，并让导出脚本生成 `buildings.json`。

建议使用清晰方案。文档和玩法都已经使用“建筑”，工程也应该逐步靠近这个语言。

## 5. 核心规则层拆解

负责目录：`packages/game-core`

### 5.1 状态与命令

文件：

- `packages/game-core/src/types.ts`
- `packages/game-core/src/state.ts`
- `packages/game-core/src/commands.ts`

需要支持的状态：

```ts
type RunPhase =
  | "build"
  | "playing"
  | "card_select"
  | "won"
  | "lost";
```

如果为了少改 UI，也可以短期保留：

```ts
type RunPhase = "ready" | "playing" | "card_select" | "won" | "lost";
```

新增或调整命令：

| 命令 | 作用 |
| --- | --- |
| `selectBuilding` | 选择要建造的建筑 |
| `placeBuilding` | 放置建筑 |
| `upgradeBuilding` | 升级建筑 |
| `startWave` | 开始下一波 |
| `selectStrategyCard` | 选择 3 张推荐卡中的 1 张 |
| `rerollRecommendations` | 暂不做，预留 |
| `applyDirectorPlan` | 应用敌方 AI 的下一波调整 |
| `resetRun` | 重置 |

初始状态建议：

| 字段 | 值 |
| --- | --- |
| `baseHp` | 10 |
| `purchasePower` | 175 |
| `waveIndex` | 0 |
| `selectedBuildingId` | `energy_core` 或 `arrow_turret` |
| `phase` | `ready` |
| `activeRecommendations` | 空数组 |
| `chosenCardTags` | 空数组 |

### 5.2 建筑系统

建议文件：

- `packages/game-core/src/systems/buildings.ts`
- 可由当前 `systems/plants.ts` 演进而来。

职责：

- 能量建筑按间隔生产购买力。
- 攻击建筑按固定方向扫描目标。
- 防御机关阻挡地面敌人。
- 建筑升级。
- 建筑血量、销毁和事件。

攻击建筑目标选择：

```text
V0.2 固定扫描右侧方向。
范围内选择距离建筑最近的敌人。
可命中地面和飞天。
若存在对空卡效果，则飞天优先。
```

第一版可以保留投射物系统，也可以直接命中。建议先保留投射物，因为当前渲染已经有 `ProjectileRenderer`。

### 5.3 敌人系统

建议文件：

- `packages/game-core/src/systems/enemies.ts`
- 可由当前 `systems/zombies.ts` 演进而来。

职责：

- 地面敌人沿路线前进。
- 地面敌人遇到防御机关停下攻击。
- 重攻敌人对建筑造成更高伤害。
- 飞天敌人不被防御机关阻挡。
- 敌人触底扣底线生命。

关键改动：

| 当前规则 | V0.2 规则 |
| --- | --- |
| 所有僵尸会咬同一行植物 | 地面敌人只被阻挡建筑停下，飞天无视阻挡 |
| dangerous 僵尸秒杀普通植物 | 重攻敌人用高 `buildingDamage` 拆建筑 |
| 基地 HP 只有 3 | V0.2 初始底线生命 10 |

### 5.4 波次系统

建议文件：

- `packages/game-core/src/systems/waves.ts`

当前波次是 `totalZombies + mix`，V0.2 需要改成分组刷怪：

```ts
type WaveEnemyGroup = {
  enemyId: string;
  count: number;
  route: "left" | "center" | "right" | "mixed";
  startSecond: number;
  interval: number;
};
```

职责：

- 根据 `enemyGroups` 在指定时间刷怪。
- 支持 Boss 第 8 秒出现。
- 波次结束时进入 `card_select`，而不是直接回到 `ready`。
- 第 5 波结束进入 `won`。

### 5.5 策略卡系统

建议新增文件：

- `packages/game-core/src/systems/cards.ts`
- `packages/game-core/src/recommender.ts`
- `packages/game-core/src/battleSnapshot.ts`

职责：

| 模块 | 职责 |
| --- | --- |
| `battleSnapshot.ts` | 每波结束生成 `BattleSnapshot` |
| `recommender.ts` | 输入 snapshot 和 card config，输出 3 张推荐卡 |
| `systems/cards.ts` | 应用玩家选择的卡牌效果 |

第一版卡牌效果建议实现优先级：

1. `emergency_base_shield`
2. `emergency_repair_all`
3. `energy_instant_power`
4. `attack_power_up`
5. `attack_speed_up`
6. `attack_anti_air_focus`
7. `defense_wall_hp_up`
8. `defense_temp_wall`

其余卡可以先接入配置和推荐，效果用 `notImplemented` 标记，避免 UI 空转。

说明：

- `pivot_direction_reset` 从 V0.2 实现范围移出，保留到 V0.3 朝向机制版本。
- V0.2 不做放置时朝向选择，也不做战中重调朝向，方向炮塔统一固定向右。

### 5.6 敌方 AI 导演

建议文件：

- `packages/game-core/src/director.ts`

当前 `director.ts` 只做隐藏数值倍率。V0.2 需要扩展成“下一波计划”：

```ts
type EnemyDirectorPlan = {
  targetWaveIndex: number;
  addedGroups: WaveEnemyGroup[];
  removedEnemyCount?: number;
  reasonTags: string[];
  previewText: string;
};
```

职责：

- 根据 `BattleSnapshot` 生成下一波轻微调整。
- 遵守 15% 压力上限。
- 不能连续两波针对同一问题。
- 结果写入下一波预告。

注意：V0.2 不让敌方 AI 实时控制敌人。

## 8. 实现顺序

建议严格按这个顺序做，避免同时改太多层。

### Step 1：内容数据对齐

目标：

- 把 `v0.2-content-tables.md` 写入 `packages/game-content/balance`。
- 生成 `buildings/enemies/cards/waves/maps` 数据。

验收：

- `pnpm content:build` 通过。
- 生成 JSON 内容与 V0.2 内容表一致。

### Step 2：运行时内容源切换

目标：

- 让 `packages/game-content/src/index.ts` 暴露 V0.2 建筑、敌人、波次。
- 暂时移除或隔离旧植物融合内容。

验收：

- `game-core` 能读取 3 个建筑、5 类敌人、5 波火山。

### Step 3：核心状态改造

目标：

- `GameState` 支持购买力、建筑、敌人、卡牌选择阶段。
- 初始 baseHp 改为 10，purchasePower 改为 175。

验收：

- 开局 HUD 能显示购买力、底线生命、3 个建筑。

### Step 4：三类建筑机制

目标：

- 能量核心生产购买力。
- 方向炮塔固定方向攻击最近敌人。
- 熔岩壁垒阻挡地面敌人。

验收：

- 只放能量核心会经济增长但不能防守。
- 只放方向炮塔能杀普通和飞天，但挡不住极速压力。
- 放熔岩壁垒能挡地面，不能挡飞天。

### Step 5：敌人与波次

目标：

- 实现普通、极速、重攻、飞天、Boss。
- 实现火山 5 波。

验收：

- 第 1 到第 5 波可以顺序开始和结束。
- Boss 波结束后进入通关。
- 底线生命为 0 时失败。

### Step 6：BattleSnapshot 和推荐卡

目标：

- 每波结束生成 `BattleSnapshot`。
- 推荐 3 张策略卡。
- 玩家选择 1 张后进入下一波准备阶段。

验收：

- 漏怪多时出现救急卡。
- 飞天压力高时出现对空卡。
- 建筑被拆时出现修复或防御卡。

### Step 7：卡牌效果

目标：

- 先实现 9 张核心效果，剩余效果可配置但暂不启用。

验收：

- 选卡后下一波能明显改变战局。
- HUD 能显示已选卡或生效状态。

### Step 8：敌方 AI 下一波调整

目标：

- 根据 `BattleSnapshot` 对下一波做有限调整。
- 预告中展示调整。

验收：

- 玩家轻松时下一波略增压。
- 玩家濒危时下一波略降压。
- 不连续针对同一弱点。

### Step 9：通关结算

目标：

- 通关后显示简单奖励。
- 失败后显示主要失败原因。

验收：

- 第 5 波胜利显示奖励。
- 失败时显示“购买力不足 / 输出不足 / 防御不足 / 对空不足 / 建筑被拆”之一。

## 11. 主要风险

| 风险 | 说明 | 建议 |
| --- | --- | --- |
| 命名冲突 | 当前代码是植物语义，V0.2 是建筑语义 | 第一阶段兼容，第二阶段重命名 |
| 内容源不一致 | `src/index.ts` 和 `balance/` 可能继续分裂 | 尽快让运行时内容从同一份 V0.2 配置生成 |
| 选卡阶段打断波次 | 当前波次结束直接回 ready | 新增 `card_select` 阶段，玩家选卡后再回 ready |
| 飞天敌人表现不清 | 当前僵尸按行走路径移动 | 渲染上提高飞行高度和阴影，规则上无视阻挡 |
| 方向攻击交互复杂 | 放置时选择方向可能拖慢实现 | V0.2 固定向右，之后再加方向选择 UI |
| 15 张卡效果过多 | 一次实现全部容易失控 | 先实现 9 张核心效果，其余只进入配置和推荐 |

## 12. V0.2 工程完成标准

1. 内容表能通过 `content:build` 生成配置。
2. 游戏使用 V0.2 的 3 个建筑、5 类敌人、5 波火山内容。
3. 玩家能完整打一局并进入胜利或失败。
4. 每波结束生成 `BattleSnapshot`。
5. 每波结束推荐 3 张策略卡，并能选择 1 张。
6. 至少 9 张卡有实际效果。
7. 敌方 AI 只调整下一波，并显示在预告中。
8. UI 能表达购买力、底线生命、建筑状态、波次、卡牌推荐和结算。
